# Plan: `pi-minimax-m3` — Portable Extension for MiniMax-M3 Passive Caching

## Goal

Convert the upstream fix in `pi-mono@b85b91c9` ("route MiniMax-M3 to openai-completions for passive caching") into a **standalone pi extension package** that:

1. Registers MiniMax-M3 on its **OpenAI-compatible** endpoint (`/v1/chat/completions`), where passive/automatic caching actually works.
2. Sets the **`skipThinkingBlock`** compat flag so M3's thinking content (sent twice — once in `reasoning_content`, once in `content` as `<think>…</think>` markers) is not duplicated.
3. Reuses the user's existing **`MINIMAX_API_KEY`** / **`MINIMAX_CN_API_KEY`** env vars, so no new auth setup is needed.
4. Lives outside the pi-mono core so it can be `pi install git:…`'d, versioned, and shipped independently of pi releases.

---

## Why a separate extension (not a core PR)

The upstream fix is correct, but the way the user encounters MiniMax-M3 is brittle:

- It's baked into `packages/ai/src/models.generated.ts` and regenerated by `scripts/generate-models.ts`, so the fix only lives in a new pi release.
- Users on older pi versions stay broken (and silently over-billed) until they upgrade.
- Anyone maintaining a custom pi fork has to re-apply the patch on every regenerate.

A pi extension sidesteps all three: it's a single TypeScript file (or small package), installable via `pi install`, and works on any pi version that supports `registerProvider` with per-model `compat` flags.

---

## Approach

`pi.registerProvider()` has two modes:

| Call | Effect |
|---|---|
| `registerProvider(name, { baseUrl, headers })` — **no `models`** | Overrides the URLs/headers of all existing models for that provider |
| `registerProvider(name, { baseUrl, apiKey, api, models })` — **with `models`** | **Replaces** every existing model for that provider |

We need M3 to use `openai-completions` + the M3 base URL + the `skipThinkingBlock` compat, but we **must not clobber the built-in `minimax`/`minimax-cn` M2.x models**. The "override" mode would lump M2.x onto the OpenAI endpoint too. The "replace" mode would wipe M2.x.

**Therefore: register a new provider** with a distinct name, exposing only the M3 model under the OpenAI-compatible endpoint. Users opt in by switching provider in `/model` (or in their config).

### Naming

| Provider name | Region | Base URL |
|---|---|---|
| `minimax-m3` | Global | `https://api.minimax.io/v1` |
| `minimax-cn-m3` | China | `https://api.minimaxi.com/v1` |

Model ID stays `"MiniMax-M3"` in both — same name upstream uses — so the extension is recognisable and replaceable once pi-mono ships the fix.

### Distinguishability from the broken built-in

In `/model` the user will see **two** `MiniMax-M3` entries side by side. To make the difference obvious, the extension registers them with a suffixed name:

| Provider | Model ID | Display name |
|---|---|---|
| `minimax` (built-in) | `MiniMax-M3` | `MiniMax-M3` |
| `minimax-m3` (extension) | `MiniMax-M3` | `MiniMax-M3 (passive cache)` |

A short `description` field is not currently supported on `ProviderModelConfig`, so the display name is the disambiguator. (If pi adds `description` to `ProviderModelConfig` later, we can move the suffix there without a breaking change.)

## Constraints & Edge Cases

| Concern | Resolution |
|---|---|
| `skipThinkingBlock` is on `OpenAICompletionsCompat` in `@earendil-works/pi-ai`. The extension must import it from there. | `import type { OpenAICompletionsCompat } from "@earendil-works/pi-ai"` — re-exported via `packages/ai/src/index.ts`. |
| `registerProvider` with `models` **replaces** existing models for the provider. | Use new provider names (`minimax-m3`, `minimax-cn-m3`) that don't collide. |
| Cost / context / maxTokens values must match upstream. | Hard-code the values from `models.generated.ts:5617-5638` (and `5671-5692` for cn). |
| Same `apiKey` env vars as the built-in (`MINIMAX_API_KEY` / `MINIMAX_CN_API_KEY`). | Reuse them verbatim — no new env var, no `/login` flow needed. |
| Built-in `minimax` provider still claims `MiniMax-M3` under `provider=minimax`. Users who pick that one stay broken. | Document in README; the fix is opt-in. (Auto-overriding would risk breaking M2.x or other models under the same provider.) |
| `compat.supportsCacheControl` (the new `anthropic-messages` flag) is **not** needed for the OpenAI-compat endpoint, so we don't need to set it. | Out of scope for the extension. |
| `cacheControlFormat: undefined` is the default for `openai-completions` — no explicit `cacheControl` markers are sent. | Don't set `cacheControlFormat` at all. |
| `reasoning: true` because M3 emits thinking. | Set on the model. |
| `input: ["text", "image"]` — M3 supports images. | Set on the model. |
| `maxTokensField: "max_tokens"` because M3's OpenAI-compat endpoint expects the legacy field. | Set in `compat`. |
| `supportsStore: false`, `supportsDeveloperRole: false`, `supportsReasoningEffort: false`, `supportsStrictMode: false`, `supportsLongCacheRetention: false` — all match upstream's M3 compat block. | Set in `compat`. |

### T2. `index.ts` — register the two providers

**File:** `index.ts`

```typescript
import type { ExtensionAPI } from "@earendil-works/pi-coding-agent";
import type { OpenAICompletionsCompat } from "@earendil-works/pi-ai";

// Mirror of packages/ai/src/models.generated.ts for the two M3 entries,
// routed to /v1/chat/completions with passive caching and the
// skipThinkingBlock compat flag (upstream b85b91c9).
const M3_COMPAT: OpenAICompletionsCompat = {
    supportsStore: false,
    supportsDeveloperRole: false,
    supportsReasoningEffort: false,
    maxTokensField: "max_tokens",
    supportsStrictMode: false,
    supportsLongCacheRetention: false,
    skipThinkingBlock: true,
};

const M3_COST = { input: 0.6, output: 2.4, cacheRead: 0.12, cacheWrite: 0 };
const M3_LIMITS = { contextWindow: 512000, maxTokens: 128000 };

function makeM3Model(suffix: string) {
    return {
        id: "MiniMax-M3",
        name: `MiniMax-M3 (passive cache${suffix})`,
        api: "openai-completions" as const,
        reasoning: true,
        input: ["text", "image"] as ("text" | "image")[],
        cost: M3_COST,
        ...M3_LIMITS,
        compat: M3_COMPAT,
    };
}

export default function (pi: ExtensionAPI) {
    // Global region
    pi.registerProvider("minimax-m3", {
        baseUrl: "https://api.minimax.io/v1",
        apiKey: "$MINIMAX_API_KEY",
        api: "openai-completions",
        models: [makeM3Model("")],
    });

    // China region
    pi.registerProvider("minimax-cn-m3", {
        baseUrl: "https://api.minimaxi.com/v1",
        apiKey: "$MINIMAX_CN_API_KEY",
        api: "openai-completions",
        models: [makeM3Model(" — CN")],
    });
}
```

**Acceptance:** `tsc --noEmit index.ts` passes (resolves the two packages via the user's installed pi at `~/.pi/agent/npm` / `~/.pi/agent/git`, or hoisted `node_modules`).

**Notes:**

- Factory is **synchronous** — no async work needed. Model data is static.
- Provider names `minimax-m3` / `minimax-cn-m3` avoid collision with built-in `minimax` / `minimax-cn`.
- `$MINIMAX_API_KEY` uses pi's standard env-var interpolation (same syntax as `models.json`); user keeps their existing env var.

## Risks & Mitigation

| Risk | Mitigation |
|---|---|
| `pi.registerProvider` semantics change in a future pi release | Pin a minimum pi version in README; surface breakage in `index.ts` with a clear error if a future compat field vanishes. |
| Built-in `minimax` provider also lists `MiniMax-M3` — users pick the wrong one | Suffix the display name; call it out in README; plan T9 covers retirement. |
| Upstream renames or removes the model ID | Bump major version of the extension; README notes the upstream `b85b91c9` commit as the source of truth. |
| The `$ENV_VAR` interpolation in `apiKey` doesn't work the same way for extension-registered providers as for `models.json` | Reuse the exact syntax already used in `custom-provider.md` ("apiKey literal, env interpolation, or !command") and verify via `pi --list-models` (T7). |
| `OpenAICompletionsCompat` loses `skipThinkingBlock` in a future `@earendil-works/pi-ai` release | The compat object is structurally typed; passing an unknown field is a no-op for older pi versions. The extension would silently lose thinking-dedup, which is exactly the original bug — flag in README and add a `console.warn` if `compat.skipThinkingBlock` is dropped. |
